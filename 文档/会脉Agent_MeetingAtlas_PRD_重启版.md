# 会脉 Agent / MeetingAtlas PRD

## 0. 文档信息

| 项目 | 内容 |
|---|---|
| 产品名称 | 会脉 Agent / MeetingAtlas |
| 产品定位 | 基于飞书会议纪要触发的个人执行闭环与主题知识库 Agent |
| 核心触发 | 飞书会议结束后，会议纪要 / 妙记 / 转写文本生成 |
| 核心能力 | 提取个人待办、生成日程草案、通过飞书卡片确认、聚合多次关联会议、创建并维护主题知识库 |
| 技术策略 | 强依赖飞书 CLI，配合 OpenClaw / 自研轻量多 Agent 编排 |
| MVP 目标 | 跑通“会后识别待办和日程 -> 用户确认 -> 创建任务/日程 -> 判断是否归入主题知识库 -> 用户确认 -> 创建或更新知识库”的完整 Demo |
| 版本 | v2.0 重启版 PRD |
| 输入依据 | 赛题公开文档、开赛直播转写、团队新录音讨论、旧版 ProjectPilot PRD |

### 0.1 这次重启版和旧方案的区别

旧方案更像“项目管理专家 Agent”，重点是项目立项、任务池、进度、风险、Dashboard。本版重新收敛为“会议纪要驱动的知识整合与分发 Agent”。

本版不再把“项目立项”作为第一触发，不再试图全量监听群聊、邮件和所有协作信息，而是把会议纪要作为最可靠、最可控、最容易解释的触发点。产品核心从“管理项目”变为“让会议产生的执行事项和知识自动沉淀，但所有高影响动作都先让用户确认”。

一句话：

> 开完会以后，Agent 不只是总结会议，而是把个人要做什么、什么时候做、这场会属于哪个主题、应该沉淀到哪里，都变成可确认、可修改、可追溯的飞书协作动作。

## 1. 背景与问题

### 1.1 赛题背景

企业和团队内部的有效信息往往散落在文档、会议纪要、群聊、任务、邮件、表格等多个飞书产品中。通用搜索和问答要求用户知道“去哪里搜”和“搜什么关键词”，但实际工作中，很多知识是在会议中被口头确认、在聊天中被补充、在文档和表格中被沉淀的。用户并不总是知道某个信息是否已经存在，也不总是知道它属于哪个项目或主题。

本赛题要求的不只是被动问答，而是能够在合适时机，以卡片、报告、CLI 提示等形态主动服务用户的知识产物。会议后自动提取 Action Items、转化为任务并附带知识库链接，是赛题中非常自然的落地方向。

### 1.2 团队重新收敛后的判断

团队讨论后形成了几个关键判断：

1. **全量收纳上下文会非常危险。** 如果所有群聊、邮件、文档都被无差别收进记忆，Agent 很容易越用越臃肿，也会把“顺手点咖啡”“借一下机器”这类小事误判成任务、日程或项目。
2. **会议纪要是更稳定的触发点。** 会议天然包含主题、参会人、上下文、结论和待办，比随机群聊更适合做结构化抽取。
3. **知识库比普通云盘文件夹更优雅。** 因为知识库可以围绕一个主题或目标组织文字内容、来源链接、会议总结和转写记录，也更适合做后续问答。
4. **必须有人类确认层。** 待办、日程、知识库归档都不应该默认自动执行；Agent 应该先生成草案，再通过飞书卡片让用户添加、修改或拒绝。
5. **搜索和来源聚合要谨慎。** 可以利用飞书搜索、文档、邮件、IM、表格等来源，但不应该把所有内容直接塞进模型上下文；需要先候选、再确认、再归档。

### 1.3 当前用户痛点

#### 对参会者

- 会后知道“好像有事情要做”，但不知道是否被正式记录。
- 会议纪要里有 Action Items，但要手动复制到任务或日历。
- 会议上提到的后续会议、评审、回访、调研安排容易漏掉。
- 同一主题开了多场会后，很难回忆每场会分别讨论了什么。

#### 对会议组织者 / 负责人

- 需要手动提醒每个人认领待办。
- 多次会议之间的关系需要人工整理。
- 项目或主题知识库经常没人维护，最终变成散乱会议纪要列表。
- 会议纪要、文档、邮件、IM、表格之间没有统一视图。

#### 对后续知识使用者

- 想了解某个主题时，需要翻多场会议、多个文档和聊天记录。
- 不知道当前进度、关键结论、未解决问题分别在哪里。
- 通用搜索返回结果多，但缺少“整体目标 - 整体分析 - 当前进度”的高密度结构。

## 2. 产品定位

### 2.1 一句话定位

会脉 Agent 是一个在飞书中以会议纪要为触发条件，自动生成个人待办、日程草案，并将多次相关会议沉淀为主题知识库的智能知识整合与分发 Agent。

### 2.2 产品不是做什么

会脉 Agent 不是：

- 不是另一个飞书会议总结功能。飞书已有会议总结，本产品重点在“总结之后的执行和沉淀”。
- 不是全量企业搜索引擎。它只围绕被确认的会议主题和知识库工作。
- 不是无脑 Todo Bot。所有待办、日程和知识库更新都先生成草案，再由用户确认。
- 不是全自动监听一切聊天和邮件的记忆系统。MVP 只以会议纪要为主动触发，其他来源以用户授权、会议引用和候选确认方式进入。
- 不是替代飞书知识问答。它更像是一个自动维护主题知识库的前置整理层。

### 2.3 核心价值

1. **把会后执行事项变成闭环。** 从会议转写中提取个人待办和日程草案，让用户一键确认创建。
2. **把多场会议变成知识库。** 自动识别同一主题下的会议，建议创建或更新主题知识库。
3. **把信息源变成可追溯证据。** 文档、邮件、IM、Excel、会议纪要等来源以链接和摘要形式进入知识库。
4. **把主动服务控制在安全边界内。** Agent 主动建议，但不擅自创建高影响内容。
5. **让知识库持续生长。** 后续类似会议在用户确认后自动并入，不断更新整体目标、分析、进度和会议记录。

### 2.4 产品原则

| 原则 | 说明 |
|---|---|
| 会议优先 | MVP 只以会议纪要作为主动触发，避免随机群聊误触发。 |
| 人类确认 | 待办、日程、知识库创建和归档都必须先通过卡片确认。 |
| 个人先行 | 待办和日程优先发给对应个人确认，而不是直接在群里公开分派。 |
| 来源可追溯 | 每一条知识更新都要能回到会议、文档、邮件、IM 或表格来源。 |
| 上下文克制 | 长文本和大表格不直接进总控 Agent，上下文按需读取。 |
| 结构化沉淀 | 知识库不只是堆会议纪要，而是维护目标、分析、进度、单会总结和转写记录。 |
| CLI-first | 飞书能力优先通过飞书 CLI 调用，由 Agent 以命令行方式组合执行。 |

## 3. 目标用户与使用场景

### 3.1 目标用户

| 用户类型 | 主要诉求 | 典型痛点 |
|---|---|---|
| 普通参会者 | 会后知道自己要做什么，快速加入任务或日程 | 会议待办分散，手动整理麻烦 |
| 会议组织者 | 确保会议结论和待办有人跟进 | 纪要有了，但执行没闭环 |
| 项目 / 主题负责人 | 多次会议后形成可持续更新的主题知识库 | 会议越多，信息越散 |
| 调研型用户 | 把多场访谈、调研、评审沉淀为结构化知识 | 每次采访都有价值，但难以汇总成整体分析 |
| 后续加入成员 | 快速理解某个主题的目标、现状和关键资料 | 需要翻历史会议和文档 |

### 3.2 MVP 首选场景

**飞书 AI 校园挑战赛团队内部协作。**

原因：

- 团队会频繁开会讨论方案、开发、路演和评审。
- 会议纪要、文档、群聊和任务都在飞书中。
- 项目周期短，Demo 能在两周内展示完整闭环。
- 赛题强调“主动知识服务”和“效果验证”，适合用卡片确认率、任务创建时间、知识库命中率证明价值。

### 3.3 示例场景：无人机操作方案调研

用户需要调研一个无人机操作方案，过程中会产生多次采访或讨论会议：

- 第一次：采访业务方，了解无人机使用目标和限制。
- 第二次：采访操作员，了解实际流程和痛点。
- 第三次：和技术团队讨论设备、权限、风险和实施方案。

会脉 Agent 应该能做到：

1. 每次会后提取参会者自己的待办和后续日程。
2. 当发现多次会议都围绕“无人机操作方案”时，建议创建主题知识库。
3. 用户确认后，创建知识库结构：整体目标、整体分析、当前进度、单个会议总结、会议转写记录、关联资料。
4. 后续类似会议再次结束后，发卡片询问是否加入该知识库。
5. 相关文档、邮件、IM、Excel 等来源先以候选资料卡片呈现，用户确认后再归档。

## 4. 用户路径

## 4.1 路径 A：会后生成个人待办

### 触发

飞书会议结束，会议纪要 / 妙记 / 转写文本可用。

### 系统动作

1. Event Listener 收到“会议纪要生成”事件。
2. Meeting Intake Agent 读取会议元信息和转写文本。
3. Meeting Extraction Agent 提取明确 Action Items。
4. Personal Action Agent 根据参会人和发言上下文将待办归属到具体用户。
5. Card Agent 给对应用户发送待办确认卡。

### 用户看到的卡片

标题：这场会里可能有 2 个属于你的待办

内容：

- 待办 1：整理无人机操作流程现状，输出一页流程图。
  - 推荐负责人：你
  - 截止时间：本周五 18:00
  - 来源：会议转写 12:30 - 13:05
  - 置信度：0.86
- 待办 2：确认无人机试飞场地权限。
  - 推荐负责人：你
  - 截止时间：待确认
  - 来源：会议转写 18:10 - 18:45
  - 置信度：0.72

操作：

- 添加到飞书任务
- 修改后添加
- 不是我的
- 拒绝
- 稍后提醒

### 成功结果

用户点击确认后，系统创建飞书任务，并把任务链接写回会议记录和知识库候选区。

## 4.2 路径 B：会后生成个人日程

### 触发

会议转写中出现明确的时间、参会人和后续安排，例如：

- “下周二下午再约一次评审。”
- “周五 10 点和操作员再做一次访谈。”
- “明天上午安排 30 分钟同步权限问题。”

### 系统动作

1. Calendar Agent 从会议中提取日程草案。
2. 判断这是“要创建日历事件”还是“只是任务截止时间”。
3. 如果信息完整，给相关用户发送日程确认卡。
4. 如果信息不完整，卡片中标注待确认字段。

### 判断规则

| 内容类型 | 处理方式 |
|---|---|
| “周五前完成方案” | 创建任务截止时间，不创建日程 |
| “周五 10 点开评审会” | 创建日程草案 |
| “找个时间聊一下” | 不创建日程，只生成待确认建议 |
| “下周约客户访谈”但无时间 | 生成日程草案，时间字段待确认 |

### 用户操作

- 添加到日历
- 修改时间 / 参会人 / 标题
- 拒绝
- 转为待办

## 4.3 路径 C：多次会议后创建主题知识库

### 触发

系统在多次会议中发现高度相关主题，例如会议标题、关键词、参会人、文档链接、待办内容都指向“无人机操作方案”。

### 系统动作

1. Topic Clustering Agent 计算当前会议和历史会议的关联度。
2. 若同一主题已有知识库，则建议加入现有知识库。
3. 若没有知识库，但至少两场会议高度相关，则建议创建新知识库。
4. Knowledge Curator Agent 生成知识库结构草案。
5. Card Agent 发出知识库确认卡。

### 用户看到的卡片

标题：是否为“无人机操作方案”创建主题知识库？

内容：

我发现最近 3 场会议都与“无人机操作方案”相关，且都提到了“操作流程”“场地权限”“风险控制”。建议创建一个主题知识库，用于沉淀目标、当前进度、会议总结和关联资料。

将加入的会议：

- 4 月 23 日：无人机方案初步访谈
- 4 月 25 日：无人机操作员访谈
- 4 月 28 日：无人机实施风险评审

默认结构：

- 整体目标
- 整体分析
- 当前进度
- 单个会议总结
- 会议转写记录
- 关联资料

操作：

- 创建知识库
- 修改名称 / 结构
- 只加入当前会议
- 暂不创建
- 不再提醒此主题

## 4.4 路径 D：后续类似会议自动建议加入知识库

### 触发

新会议结束后，Topic Clustering Agent 发现它和某个已有知识库高度相关。

### 系统动作

1. 匹配已有知识库。
2. 生成“加入知识库”草案。
3. 用户确认后追加本次会议摘要和转写记录。
4. 更新整体分析和当前进度。
5. 在知识库变更记录中保留本次更新。

### 默认策略

- 默认仍需要用户确认。
- 用户连续 3 次确认某类会议加入同一知识库后，可以建议开启“默认加入但每次通知”的半自动模式。
- 即使开启半自动模式，涉及任务创建、日程创建、跨知识库合并、敏感来源归档，仍必须确认。

## 4.5 路径 E：知识库聚合文档、邮件、IM、Excel 等资料

### 触发

以下情况会触发候选资料搜索：

- 会议中出现文档链接、表格链接或邮件主题。
- 会议中多次提到某个关键词、客户名、项目名或资料名。
- 用户在知识库中点击“查找相关资料”。
- 用户手动转发文档、邮件或消息给 Agent。

### 系统动作

1. Source Retrieval Agent 根据会议摘要、关键词和知识库主题调用飞书搜索能力。
2. 返回候选资料列表，不直接写入知识库。
3. 对每个候选资料生成摘要、命中原因和来源链接。
4. 用户确认后归档到知识库“关联资料”区。

### 候选资料卡片

标题：发现 4 个可能与“无人机操作方案”相关的资料

候选资料：

1. 文档：无人机试飞安全规范
   - 命中原因：标题包含“无人机”“安全”，会议中提到“安全规范”
   - 推荐归档位置：关联资料 / 规范资料
2. 邮件：关于试飞场地权限确认
   - 命中原因：发件人为场地负责人，主题包含“权限确认”
   - 推荐归档位置：关联资料 / 权限与审批
3. IM 消息：操作员反馈试飞限制
   - 命中原因：会议中提到该群讨论
   - 推荐归档位置：关联资料 / 操作反馈
4. Excel / 多维表格：设备清单与状态
   - 命中原因：会议中提到“设备清单”
   - 推荐归档位置：关联资料 / 设备与资源

操作：

- 全部归档
- 逐条确认
- 修改归档位置
- 忽略

## 5. 产品范围

### 5.1 P0 必须完成

1. 会议纪要 / 转写文本输入。
2. 自动提取会议 Action Items。
3. 自动提取后续日程草案。
4. 通过飞书卡片询问用户添加、修改或拒绝。
5. 用户确认后创建飞书任务。
6. 用户确认后创建飞书日历事件。
7. 自动判断会议与已有主题知识库的关联。
8. 多次相关会议后，建议创建主题知识库。
9. 用户确认后创建知识库基础结构。
10. 用户确认后将会议摘要和转写记录加入知识库。
11. 使用多维表格或本地状态表记录会议、待办、日程、知识库、确认请求和来源链接。
12. 所有飞书操作通过飞书 CLI 封装调用。

### 5.2 P1 强烈建议完成

1. 飞书文档、知识库、IM、邮件、Excel / 多维表格的候选资料搜索。
2. 候选资料确认卡。
3. 知识库问答能力，只回答当前知识库范围内的问题。
4. 当前进度自动更新。
5. 决策记录、风险记录、变更记录自动沉淀。
6. 相似会议自动加入知识库的半自动偏好。
7. 会议前 Brief：基于已有知识库推送会前背景资料。
8. 来源引用和证据片段展示。

### 5.3 P2 有时间再做

1. 全局多知识库关系图谱。
2. 用户个人跨会议待办总览。
3. 自动识别知识库过期内容。
4. 复杂权限策略和跨部门授权。
5. 对大型 Excel / 多维表格进行局部结构化分析。
6. 多 Agent 自主协商和任务委派。
7. 长周期团队洞察周报。

### 5.4 非目标

MVP 不做：

- 不全量监听所有群聊消息。
- 不全量扫描用户全部邮箱。
- 不无确认地创建任务、日程或知识库。
- 不做删除任务、删除文档、删除知识库等高风险操作。
- 不承诺替代飞书官方会议总结或知识问答。
- 不把所有历史资料直接塞入模型上下文。
- 不处理图片、视频、语音附件的复杂理解，除非已转写为文本。

## 6. 功能需求

## 6.1 会议纪要接入

### 功能说明

系统在会议结束后获取会议纪要、妙记摘要和转写文本。若接口权限不足，MVP 允许用户手动粘贴转写文本或上传测试纪要。

### 输入

- meeting_id
- meeting_title
- meeting_time
- organizer
- participants
- minutes_url
- transcript_text
- ai_summary_from_feishu，可选
- linked_docs，可选
- calendar_event_id，可选

### 输出

- MeetingRaw 记录
- MeetingNormalized 记录
- 会议处理状态：pending / extracted / confirmed / archived / failed

### 处理规则

1. 会议标题为空时，使用日历标题或转写中高频主题生成候选标题。
2. 参会人缺失时，不强行推断负责人，只标记 unknown。
3. 转写文本过长时，分段处理，先生成段落摘要和证据片段。
4. 所有后续抽取结果都必须绑定 source_span 或 source_url。

### 验收标准

- 能处理一份完整会议转写文本。
- 能识别会议标题、时间、参会人和来源链接。
- 能把长转写拆分为可被下游 Agent 使用的结构化片段。

## 6.2 会议理解与结构化抽取

### 功能说明

从会议转写中提取以下结构：

- 会议摘要
- 关键结论
- 个人 Action Items
- 日程草案
- 主题关键词
- 风险 / 阻塞
- 决策记录
- 资料引用
- 可能关联的知识库线索

### 抽取原则

1. **只提取明确可执行事项。** “可以看看”“有机会研究”不应直接成为任务。
2. **不确定字段标记为 null。** 不编造负责人、日期和会议时间。
3. **区分任务截止时间和日历事件。** “周五前完成”是任务截止时间，“周五十点开会”才是日历事件。
4. **保留证据。** 每个待办、日程和结论都要包含转写片段或时间戳。
5. **置信度透明。** 置信度低的结果只能作为待确认草案。

### 输出示例

```json
  "action_items": [
    {
      "title": "整理无人机现有操作流程",

      "due_date": "2026-05-01",
      "priority": "P1",

    {
      "title": "无人机操作员访谈",
```

## 6.3 个人待办生成

### 功能说明

Personal Action Agent 将会议 Action Items 转换成用户自己的待办草案，并通过飞书卡片发送给对应用户。

### 待办字段

| 字段 | 说明 |
|---|---|
| action_id | 内部 ID |
| title | 待办标题 |
| description | 背景说明 |
| owner | 推荐负责人 |
| collaborators | 协作者 |
| due_date | 截止时间 |
| priority | P0 / P1 / P2 |
| source_meeting_id | 来源会议 |
| source_span | 来源转写片段或时间戳 |
| suggested_reason | 为什么认为这是该用户的待办 |
| confidence | 置信度 |
| status | draft / confirmed / rejected / created |

### 负责人判断依据

- 会议中被明确点名。
- 用户自己说“我来做”。
- 该用户在会议中负责对应模块或话题。
- 历史同主题知识库中该用户多次负责类似事项。
- 若无法判断，则 owner = null，发给会议组织者确认。

### 卡片交互

用户可以：

- 添加到飞书任务。
- 修改标题、负责人、截止时间、优先级后添加。
- 标记“不是我的”，转给组织者或另一个人确认。
- 拒绝该待办。
- 稍后提醒。

### 验收标准

- 能从会议中提取明确待办。
- 负责人和截止时间不确定时不乱编。
- 用户确认前不创建飞书任务。
- 用户修改后创建的任务应使用修改后的字段。

## 6.4 日程草案生成

### 功能说明

Calendar Agent 从会议转写中提取后续会议、访谈、评审、同步会等日程草案。

### 日程字段

| 字段 | 说明 |
|---|---|
| calendar_draft_id | 内部 ID |
| title | 日程标题 |
| start_time | 开始时间 |
| end_time / duration | 结束时间或时长 |
| participants | 参会人 |
| location | 地点或线上会议 |
| agenda | 议程 |
| related_topic_id | 关联知识库 |
| source_meeting_id | 来源会议 |
| missing_fields | 待确认字段 |
| confidence | 置信度 |
| status | draft / confirmed / rejected / created |

### 日程和任务的区分

- 有明确会议、同步、访谈、评审、沟通动作，并出现时间或参与人，优先生成日程。
- 只有交付物和截止时间，生成任务，不生成日程。
- 表达模糊时，只生成“建议安排日程”的卡片，不直接创建。

### 验收标准

- 能正确识别后续会议类表达。
- 能将“截止时间”与“日历事件”区分开。
- 用户确认前不创建日历事件。

## 6.5 飞书确认卡片

### 功能说明

卡片是所有高影响动作的安全确认层。Agent 只生成草案，用户通过卡片添加、修改或拒绝。

### 卡片类型

| 卡片 | 使用场景 | 默认接收人 |
|---|---|---|
| 个人待办确认卡 | 会议后识别到个人待办 | 推荐负责人 |
| 日程确认卡 | 会议后识别到后续会议或访谈 | 组织者和相关参与人 |
| 知识库创建卡 | 多场会议形成新主题 | 会议组织者 / 主题负责人 |
| 知识库归档卡 | 当前会议可能属于已有知识库 | 会议组织者 / 主题负责人 |
| 资料归档卡 | 发现相关文档、邮件、IM、Excel | 知识库负责人 |
| 更新摘要卡 | 知识库更新完成 | 相关群或负责人 |

### 卡片通用状态机

```text
Draft -> Sent -> UserEdited -> Confirmed -> Executed -> Logged
                   |              |
                   v              v
                Rejected       Failed / Retry
```

### 卡片设计要求

1. 展示“Agent 发现了什么”。
2. 展示“为什么这样建议”。
3. 展示来源链接或证据片段。
4. 展示置信度和待确认字段。
5. 允许修改核心字段。
6. 操作按钮必须包含确认、修改、拒绝。
7. 执行完成后反馈结果和链接。

### 关键安全要求

- 批量创建任务必须确认。
- 创建日程必须确认。
- 创建知识库必须确认。
- 将会议加入知识库必须确认。
- 归档邮件、IM、文档、表格等资料必须确认。
- 删除操作 MVP 不开放。

## 6.6 主题聚类与知识库匹配

### 功能说明

Topic Clustering Agent 判断一场会议是否属于某个已有主题知识库，或是否与历史会议组成一个新主题。

### 输入信号

- 会议标题相似度。
- 会议摘要相似度。
- 关键词重叠度。
- 参会人重叠度。
- 待办和日程主题相似度。
- 文档、表格、邮件链接重叠度。
- 用户历史确认行为。
- 显式项目名、客户名、目标名、调研对象名。

### 匹配策略

| 分数区间 | 动作 |
|---|---|
| score < 0.60 | 不触发，不打扰用户 |
| 0.60 <= score < 0.78 | 进入候选观察队列，不推送 |
| 0.78 <= score < 0.90 | 发送轻量确认卡：是否加入已有知识库 |
| score >= 0.90 | 发送强确认卡：高度相关，但仍需用户确认 |

### 创建新知识库的条件

满足任一条件即可建议创建：

1. 至少 2 场会议与同一主题高度相关，且尚无知识库。
2. 当前会议明确提出“需要整理成知识库 / 调研档案 / 项目资料”。
3. 用户在卡片或群聊中显式要求创建。
4. 已有多个待办、资料和后续会议指向同一目标。

### 验收标准

- 能识别同一主题下的多场会议。
- 不会因一次普通闲聊或弱相关会议直接创建知识库。
- 用户拒绝后，应记录 suppression_key，避免重复打扰。

## 6.7 主题知识库创建与结构

### 功能说明

Knowledge Curator Agent 在用户确认后创建主题知识库，并按固定结构组织内容。

### 推荐知识库结构

```text
主题知识库：无人机操作方案
├── 00 首页 / 总览
├── 01 整体目标
├── 02 整体分析
├── 03 当前进度
├── 04 关键结论与决策
├── 05 待办与日程索引
├── 06 单个会议总结
│   ├── 2026-04-23 无人机方案初步访谈
│   ├── 2026-04-25 操作员访谈
│   └── 2026-04-28 实施风险评审
├── 07 会议转写记录
│   ├── 2026-04-23 转写
│   ├── 2026-04-25 转写
│   └── 2026-04-28 转写
├── 08 关联资料
│   ├── 文档
│   ├── 邮件
│   ├── IM 消息
│   ├── Excel / 多维表格
│   └── 其他会议纪要
├── 09 风险、问题与待验证假设
└── 10 变更记录
```

### 首页内容

- 主题名称。
- 一句话目标。
- 背景说明。
- 当前进度。
- 本周新增信息。
- 关键结论。
- 未解决问题。
- 最近会议。
- 关联待办。
- 关联日程。
- 关联资料。

### 整体目标

由多次会议中反复出现的目标和用户确认的目标生成。若目标不明确，标记“待确认”。

### 整体分析

按照主题总结：

- 背景与动机。
- 已知事实。
- 用户需求。
- 约束条件。
- 方案选项。
- 风险与不确定性。
- 推荐下一步。

### 当前进度

结合已确认的待办、日程、会议结论和知识库更新生成，不允许模型凭空判断。进度可以分为：

- 未启动。
- 调研中。
- 方案设计中。
- 执行中。
- 验证中。
- 已完成。

### 单个会议总结

每场会议进入知识库时，生成一页或一个区块：

- 会议时间。
- 参会人。
- 会议摘要。
- 关键结论。
- 待办。
- 日程。
- 风险。
- 资料引用。
- 原始会议纪要链接。

### 会议转写记录

转写记录单独存放，避免污染知识库首页。首页和分析页只引用摘要和关键片段。

### 验收标准

- 能自动创建上述基础结构。
- 能将多场会议分别归档到“单个会议总结”和“会议转写记录”。
- 首页和整体分析能随新会议增量更新。
- 每次更新都写入变更记录。

## 6.8 关联资料搜索与归档

### 功能说明

Source Retrieval Agent 根据会议和知识库主题搜索相关资料，生成候选资料列表。候选资料必须用户确认后才能归档。

### 资料类型

| 类型 | 处理方式 |
|---|---|
| 飞书文档 | 读取标题、摘要、链接；必要时读取正文摘要 |
| 知识库页面 | 读取标题、路径、摘要和链接 |
| IM 消息 | 只读取搜索结果片段和链接；默认不全量保存聊天记录 |
| 邮件 | 只读取授权范围内邮件；默认保存摘要和链接，不保存完整正文 |
| Excel / 多维表格 | 读取表名、字段、选定范围或摘要，不默认读取全表 |
| 会议纪要 | 读取标题、摘要、参会人和链接 |
| 飞书任务 | 读取任务标题、负责人、状态、截止时间和链接 |

### 搜索策略

1. 从会议和知识库中抽取关键词。
2. 根据关键词调用飞书搜索或相关 CLI 能力。
3. 对结果做类型过滤、权限过滤和去重。
4. 对 Top-K 结果生成候选摘要。
5. 通过卡片让用户逐条确认。

### 归档字段

- source_id
- source_type
- title
- source_url
- summary
- why_related
- archive_section
- added_by
- confirmed_by
- created_at
- last_checked_at

### 验收标准

- 能发现至少一种外部资料来源，例如文档或表格。
- 能显示命中原因和来源链接。
- 用户确认后才写入知识库。
- 用户拒绝后不再重复推荐同一资料。

## 6.9 知识库增量更新

### 功能说明

每次确认加入新会议或资料后，Knowledge Curator Agent 以增量方式更新知识库。

### 更新内容

- 首页最近更新。
- 整体分析新增或修正。
- 当前进度更新。
- 新增单个会议总结。
- 新增会议转写记录链接。
- 新增关联资料。
- 新增待办 / 日程索引。
- 新增风险、问题或假设。
- 新增变更记录。

### 冲突处理

若新会议与旧结论冲突，例如旧结论是“先不做技术方案”，新会议决定“本周启动技术方案”，系统不应直接覆盖，而应创建变更记录：

- 变更前。
- 变更后。
- 变更来源。
- 影响范围。
- 是否需要通知相关人。

### 验收标准

- 新会议加入后，知识库首页能看到更新。
- 旧会议摘要不被覆盖。
- 冲突信息进入变更记录。
- 所有更新可追溯到来源。

## 6.10 知识库问答

### 功能说明

用户可以在知识库内询问与主题相关的问题。QA Agent 只在该知识库范围内检索，不进行全企业搜索。

### 示例问题

- “无人机操作方案现在最大的风险是什么？”
- “上次是谁负责确认场地权限？”
- “我们为什么决定先做访谈？”
- “目前有哪些未完成待办？”
- “操作员提到的限制有哪些？”

### 回答规则

1. 必须引用来源会议、文档、邮件或表格链接。
2. 找不到证据时明确说“当前知识库没有找到”。
3. 不跨知识库回答，除非用户明确要求扩展搜索。
4. 对待办、负责人、截止时间等执行信息，以已确认记录为准。

### 验收标准

- 能基于知识库资料回答至少 5 类问题。
- 回答中包含来源链接。
- 没有来源时不编造。

## 6.11 防打扰与误触发控制

### 问题来源

团队讨论中提到，很多日常消息只是“帮忙点咖啡”“借一下机器”这类顺手小事，不应该被 Agent 升级成任务、日程或知识库。因此，本产品必须把误触发控制作为核心设计，而不是事后补丁。

### 防打扰策略

1. MVP 只以会议纪要作为主动触发。
2. 低置信度事项只进入草稿，不推送。
3. 同一会议最多发送一次个人待办卡、一次日程卡、一次知识库建议卡。
4. 用户点击拒绝后，记录拒绝原因。
5. 用户点击“不再提醒此主题”后，将 suppression_key 写入表。
6. 对 IM、邮件、文档的搜索只在知识库或会议明确指向时触发。
7. 所有主动建议必须可解释：为什么触发、来源是什么、置信度多少。

### 触发阈值

| 场景 | 建议阈值 |
|---|---|
| 个人待办卡 | confidence >= 0.70 或 owner 明确 |
| 日程草案卡 | confidence >= 0.75 且有时间或可补全字段 |
| 加入已有知识库 | score >= 0.78 |
| 创建新知识库 | 至少两场会议相关，或用户显式要求 |
| 候选资料归档 | source_score >= 0.70 |

## 8. 多 Agent 架构

## 8.1 为什么采用多 Agent

单 Agent 方案容易出现三个问题：

1. 工具过多，模型不知道什么时候调用会议、任务、日历、文档、知识库、搜索或邮件能力。
2. 上下文过长，转写、资料、任务和历史知识库混在一起，容易爆炸。
3. 职责不清，难以评测“是抽取错了、匹配错了、还是执行错了”。

多 Agent 的目标不是堆复杂框架，而是把能力边界划清楚：

- 总控 Agent 只做路由和决策。
- 会议 Agent 只读会议。
- 待办 Agent 只处理 Action Items。
- 日程 Agent 只处理 Calendar Drafts。
- 知识库 Agent 只做结构化沉淀。
- 来源 Agent 只做候选资料检索。
- 卡片 Agent 只做用户交互和确认。

## 8.2 推荐架构

```text
Feishu Meeting Minutes / Transcript
        ↓
Event Listener
        ↓
PM Orchestrator
        ↓
┌────────────────────────────────────────────┐
│ Meeting Extraction Agent                   │
│ Personal Action Agent                      │
│ Calendar Agent                             │
│ Topic Clustering Agent                     │
│ Knowledge Curator Agent                    │
│ Source Retrieval Agent                     │
│ Card Interaction Agent                     │
│ State Store Agent                          │
│ QA Agent                                   │
└────────────────────────────────────────────┘
        ↓
Feishu CLI Wrapper
        ↓
Feishu Task / Calendar / Wiki / Doc / Base / IM / Mail / Search
```

## 8.3 Agent 分工

### 0. PM Orchestrator 总控 Agent

职责：

- 接收事件。
- 判断事件类型。
- 决定调用哪些子 Agent。
- 合并子 Agent 输出。
- 判断是否需要发确认卡。
- 触发执行 Agent。

限制：

- 不读取完整会议转写。
- 不读取完整知识库。
- 不直接创建任务或日程。
- 不直接扫描邮件或 IM。

### 1. Meeting Intake Agent

职责：

- 获取会议元信息。
- 获取会议纪要和转写文本。
- 处理长文本分段。
- 生成 MeetingRaw 和 MeetingNormalized。

工具：

- lark minutes / vc 相关 CLI 能力。
- lark calendar 读取会议信息。
- lark doc 读取手动粘贴文档。

### 2. Meeting Extraction Agent

职责：

- 生成会议摘要。
- 提取关键结论。
- 提取 Action Items。
- 提取日程草案。
- 提取主题关键词。
- 提取风险、决策、资料引用。

限制：

- 只输出 JSON。
- 所有字段必须有证据或 null。

### 3. Personal Action Agent

职责：

- 判断待办是否明确可执行。
- 判断推荐负责人。
- 生成个人待办草案。
- 标注置信度和证据。

工具：

- lark contact。
- lark task，仅在确认后创建任务。
- lark base，写入草案状态。

### 4. Calendar Agent

职责：

- 判断会议中是否有日程草案。
- 补全日程标题、时间、参会人、议程。
- 将模糊时间标记为待确认。
- 用户确认后创建飞书日历事件。

工具：

- lark calendar。
- lark contact。
- lark im / card。

### 5. Topic Clustering Agent

职责：

- 计算当前会议与已有知识库的相似度。
- 识别是否形成新主题。
- 生成创建知识库或加入知识库的建议。
- 管理 suppression_key，减少重复打扰。

工具：

- lark base，读取知识库和会议历史。
- 可选向量检索或本地 embedding。

### 6. Knowledge Curator Agent

职责：

- 创建知识库结构。
- 写入整体目标、整体分析、当前进度。
- 追加单个会议总结。
- 追加会议转写记录。
- 更新资料索引和变更记录。

工具：

- lark wiki。
- lark doc。
- lark base。

### 7. Source Retrieval Agent

职责：

- 根据会议和知识库关键词搜索相关资料。
- 生成候选资料列表。
- 生成命中原因和归档建议。
- 用户确认后归档。

工具：

- lark search。
- lark doc / wiki。
- lark mail。
- lark im。
- lark base / sheets。

### 8. Card Interaction Agent

职责：

- 生成飞书卡片。
- 接收卡片回调。
- 解析用户编辑字段。
- 将确认结果写入 ConfirmationRequests。
- 调用执行 Agent。

工具：

- lark im。
- 飞书卡片 / CardKit。

### 9. State Store Agent

职责：

- 维护多维表格状态。
- 写入会议、待办、日程、知识库、来源、确认请求。
- 提供统一查询接口给其他 Agent。

工具：

- lark base。
- 本地 JSON / SQLite 作为开发兜底。

### 10. QA Agent

职责：

- 在单个知识库内回答问题。
- 检索来源片段。
- 生成带来源链接的答案。
- 对无证据问题拒绝编造。

工具：

- lark doc / wiki。
- lark base。
- lark search，可选。

### 11. Evaluation Agent

职责：

- 对抽取结果和用户反馈做统计。
- 生成效果验证报告。
- 统计确认率、修改率、拒绝率和节省时间。

工具：

- lark base。
- lark doc。

## 8.4 推荐实现方式

### 阶段 1：伪多 Agent，最适合比赛 MVP

一个后端服务实现多个 Agent 函数，每个函数有独立 Prompt、输入 Schema、输出 Schema 和可调用工具。

```text
orchestrator()
meetingIntakeAgent()
meetingExtractionAgent()
personalActionAgent()
calendarAgent()
topicClusteringAgent()
knowledgeCuratorAgent()
sourceRetrievalAgent()
cardInteractionAgent()
stateStoreAgent()
qaAgent()
```

优点：

- 开发速度最快。
- 容易 Debug。
- 容易录 Demo。
- 不需要一开始搭复杂 Agent Runtime。

### 阶段 2：轻量多 Agent 模块化

将每个 Agent 独立成文件，统一接口：

```ts
interface AgentInput {
  eventType: string;
  projectId?: string;
  kbId?: string;
  payload: Record<string, unknown>;
}

interface AgentOutput {
  status: "success" | "needs_confirmation" | "failed";
  data: Record<string, unknown>;
  nextAgents?: string[];
  confirmationRequest?: ConfirmationRequest;
  errors?: string[];
}
```

### 阶段 3：真正多 Agent Runtime

如果时间充足，再考虑：

- OpenClaw 编排。
- LangGraph。
- OpenAI Agents SDK。
- Temporal / BullMQ 事件队列。
- 自研 Agent Runtime。

比赛 MVP 不建议第一天就上复杂框架。

## 9. 飞书 CLI 集成方案

## 9.1 为什么强依赖 CLI

飞书 CLI 对 Agent 更友好：

- 命令行天然是文本输入、文本输出，适合模型使用。
- Agent 可以通过 help / schema 逐步探索命令，而不是一次性把所有工具定义塞进上下文。
- 大结果可以落盘、管道过滤或用脚本处理，不必全部进入模型上下文。
- 多业务域能力可以通过 CLI 组合，例如会议、任务、日历、文档、知识库、多维表格、IM、邮件和搜索。

## 9.2 CLI Wrapper 设计

后端不应该在业务代码中散落 CLI 命令，而应该统一封装。

```ts
export async function runLarkCli(args: string[], options?: RunOptions) {
  const result = await execFileAsync("lark", args, {
    timeout: options?.timeout ?? 30000,
    env: process.env
  });

  if (result.stderr) {
    logger.warn(result.stderr);
  }

  return parseCliOutput(result.stdout);
}
```

所有 Agent 只调用工具层：

```text
tools/minutes.ts
tools/task.ts
tools/calendar.ts
tools/wiki.ts
tools/doc.ts
tools/base.ts
tools/im.ts
tools/search.ts
tools/mail.ts
```

注意：实际 CLI 命令名称必须以当前环境的 `lark --help`、`lark schema` 和官方 skill 文档为准，PRD 不写死不可验证命令。

## 9.3 需要使用的飞书能力

| 飞书能力 | 用途 | MVP 优先级 |
|---|---|---|
| IM / Bot / Card | 发送确认卡、接收用户操作 | P0 |
| Minutes / VC | 读取会议纪要和转写 | P0，如权限不足可手动输入兜底 |
| Task | 用户确认后创建个人待办 | P0 |
| Calendar | 用户确认后创建日程 | P0 |

| Mail | 授权范围内邮件归档 | P1 |
| Sheets / Base | 关联 Excel / 多维表格资料 | P1 |

## 9.4 身份与权限

| 身份 | 使用场景 | 说明 |
|---|---|---|
| Bot 身份 | 发卡片、创建知识库草案、写入公共表 | 适合团队空间动作 |
| 用户身份 | 创建个人任务、个人日程、读取个人授权资料 | 必须明确授权 |
| 混合模式 | Bot 生成草案，用户确认后以用户身份执行 | MVP 推荐 |

## 10. 系统工作流

## 10.1 工作流 1：会议后待办和日程闭环


## 11. Prompt 与结构化输出规范

## 11.1 Meeting Extraction Agent Prompt 摘要

```text
你是会议纪要分析 Agent。
你的任务是从会议转写中提取会议摘要、明确待办、日程草案、主题关键词、决策、风险和资料引用。
只输出 JSON，不要输出解释。
缺失字段使用 null，不要编造。
每个待办和日程都必须包含 evidence。
只有明确可执行事项才能成为 action_item。
区分任务截止时间和日历事件。
```

## 11.2 Topic Clustering Agent Prompt 摘要

```text
你是主题聚类 Agent。
你的任务是判断当前会议是否属于某个已有主题知识库，或是否应与历史会议组成新知识库。

输出匹配分数、匹配理由、建议动作。
不要直接创建知识库。
```

## 11.3 Knowledge Curator Agent Prompt 摘要

```text
你是知识库维护 Agent。
你的任务是把已确认的会议和资料增量写入主题知识库。
知识库必须包含整体目标、整体分析、当前进度、单个会议总结、会议转写记录和关联资料。
新内容必须保留来源链接。
如果新结论和旧结论冲突，不要覆盖，生成变更记录。
```

## 12. 飞书卡片设计

## 12.1 个人待办确认卡

### 卡片内容

- 会议标题和时间。
- 识别到的待办列表。
- 每条待办的负责人、截止时间、优先级、证据、置信度。
- 可编辑字段：标题、负责人、截止时间、优先级、备注。
- 操作：添加到任务、修改后添加、不是我的、拒绝。

### 文案示例

> 我从“无人机操作方案访谈”中识别到 2 个可能属于你的待办。你可以直接添加到飞书任务，也可以修改负责人或截止时间后再确认。

## 12.2 日程确认卡

### 卡片内容

- 日程标题。
- 推荐时间。
- 参会人。
- 议程。
- 来源证据。
- 缺失字段。
- 操作：添加到日历、修改后添加、转为待办、拒绝。

### 文案示例

> 会议中提到“下周二下午再做一次评审”，我生成了一个日程草案。当前时间字段置信度较低，你可以修改后再添加。

## 12.3 知识库创建卡

### 卡片内容

- 推荐知识库名称。
- 创建原因。
- 关联会议列表。
- 默认结构。
- 推荐负责人。

## 13. 权限、安全与隐私

### 13.1 权限原则

1. 最小权限，只申请完成当前功能所需权限。
2. 用户身份和 Bot 身份分离。
3. 默认不读取用户全部邮箱、全部群聊或全部文档。
4. 只读取用户有权限访问的内容。
5. 邮件和 IM 默认只保存摘要和链接，不保存完整正文。
6. 所有来源归档都要显示来源和命中原因。

### 13.2 操作风险分级

| 操作 | 风险等级 | 是否需要确认 |
|---|---|---|
| 读取会议纪要 | 低 | 否，前提是已授权 |
| 生成待办草案 | 低 | 否 |
| 创建飞书任务 | 中 | 是 |
| 创建日历事件 | 中 | 是 |
| 创建知识库 | 中 | 是 |
| 将会议加入知识库 | 中 | 是 |
| 归档文档 / 邮件 / IM / 表格 | 中到高 | 是 |
| 修改已有知识库分析结论 | 高 | 是或保留变更记录 |
| 删除任务 / 日程 / 知识库 | 极高 | MVP 不做 |

### 13.3 审计记录

所有确认和执行都写入 ConfirmationRequests 和 KnowledgeUpdates：

- 谁确认。
- 确认前是什么。
- 用户修改了什么。
- 执行了什么。
- 来源链接是什么。
- 执行是否成功。

### 14.3 Orchestrator 伪代码

```ts
async function orchestrator(event: FeishuEvent) {
  const normalized = normalizeEvent(event);

  if (normalized.type === "meeting_minutes_ready") {
    const meeting = await meetingIntakeAgent.run(normalized.payload);
    const extraction = await meetingExtractionAgent.run({ meeting });

    const actionDrafts = await personalActionAgent.run({ meeting, extraction });
    const calendarDrafts = await calendarAgent.run({ meeting, extraction });

    await cardInteractionAgent.sendActionCards(actionDrafts);
    await cardInteractionAgent.sendCalendarCards(calendarDrafts);

    const topicMatch = await topicClusteringAgent.run({ meeting, extraction });

    if (topicMatch.suggested_action === "ask_append") {
      await cardInteractionAgent.sendAppendKnowledgeBaseCard(topicMatch);
    }

    if (topicMatch.suggested_action === "ask_create") {
      await cardInteractionAgent.sendCreateKnowledgeBaseCard(topicMatch);
    }

    await stateStore.saveMeetingProcessingResult({
      meeting,
      extraction,
      actionDrafts,
      calendarDrafts,
      topicMatch
    });
  }

  if (normalized.type === "card_confirmed") {
    await confirmationService.execute(normalized.payload);
  }
}
```

| 场景 | 传统方式 | Agent 方式 | 目标 |

- 待办确认率。
- 待办修改率。
- 待办拒绝率。
- 日程确认率。
- 知识库创建建议采纳率。

- 修改率过高：说明抽取字段不够准，但草案仍有价值。

- 知识库采纳率高：说明多会议聚类方向有效。
- 候选资料采纳率低：说明搜索关键词或来源筛选需要优化。

## 18. 验收标准

### P0 验收

1. 输入一份会议转写，可以输出结构化会议摘要、待办和日程草案。
2. 待办和日程通过飞书卡片发送给用户。
3. 用户可以添加、修改或拒绝。
4. 用户确认后，系统能创建飞书任务和日历事件。
5. 输入多份相关会议，系统能建议创建主题知识库。
6. 用户确认后，系统能创建知识库并写入固定结构。
7. 后续会议可以被建议加入已有知识库。
8. 所有确认和执行动作都有状态记录。
9. 高风险动作不会绕过用户确认。
10. Demo 中至少展示一个主题知识库的完整更新过程。

### P1 验收

1. 能搜索并推荐至少一种外部资料来源。
2. 能通过候选资料卡确认归档。
3. 能在知识库范围内回答问题并展示来源。
4. 能生成当前进度和关键风险。

## 19. 关键风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 会议转写质量不稳定 | 待办和日程抽取错误 | 使用证据片段、置信度、确认卡兜底 |
| 误把小事变任务 | 用户反感 | 只以会议触发，设置阈值和拒绝反馈 |
| 日程和任务混淆 | 造成错误日历 | 明确判断规则，模糊事项转为待确认 |
| 知识库过度创建 | 知识库泛滥 | 至少两场相关会议或用户显式要求 |
| 关联资料噪声高 | 知识库变杂 | 候选资料必须显示命中原因并确认 |
| CLI 命令变化 | 开发卡住 | 统一 wrapper，以本地 help/schema 为准 |
| 权限不足 | 无法读取会议或创建任务 | 手动输入和模拟数据兜底，逐步申请权限 |
| 上下文爆炸 | 成本高、幻觉多 | 长文本分段，状态入 Base，Agent 按需读取 |
| 用户卡片疲劳 | 不愿点击确认 | 合并同类提醒，限制每场会卡片数量 |
| 多人确认冲突 | 任务归属混乱 | owner 确认优先，组织者可最终修正 |

## 20. 需要团队拍板的默认决策

### 决策 1：个人待办使用飞书任务还是个人 Todo？

默认建议：使用飞书任务作为 MVP 执行载体。

理由：飞书任务更适合团队协作、负责人、截止时间、状态和链接回写。若用户希望更个人化，可在 P1 支持个人 Todo 或任务私密创建。

### 决策 2：知识库使用飞书 Wiki 还是普通文档文件夹？

默认建议：使用飞书知识库 / Wiki。

理由：知识库更适合长期结构化沉淀，也更符合“整体目标 - 整体分析 - 当前进度 - 会议记录 - 资料索引”的组织方式。

### 决策 3：未来类似会议是否自动加入知识库？

默认建议：默认每次询问；连续确认后允许用户开启半自动模式。

理由：用户明确提出后续类似会议也需要在确认后加入，因此默认不绕过确认。

### 决策 4：是否接入邮件和 IM？

默认建议：MVP 不全量接入，只做候选资料来源；P1 再接入授权范围内邮件和 IM 搜索。

理由：这是误触发和隐私风险最高的部分，应先用会议纪要闭环证明价值。

## 21. 最终总结

会脉 Agent 的核心不是“再做一个会议总结”，而是把飞书会议总结之后最容易断掉的三件事补上：

1. **执行闭环。** 谁要做什么、什么时候做，通过卡片确认后进入飞书任务和日历。
2. **知识闭环。** 多次相关会议自动形成主题知识库，沉淀整体目标、整体分析、当前进度、单会总结和转写记录。
3. **来源闭环。** 文档、邮件、IM、Excel、会议纪要等资料以候选方式进入知识库，用户确认后归档，后续问答可追溯来源。

这版产品的边界非常清楚：

- 会议纪要是触发条件。
- 飞书 CLI 是能力底座。
- 飞书卡片是安全确认层。
- 多维表格是结构化状态中枢。
- 知识库是长期沉淀空间。
- 多 Agent 是上下文和职责隔离方式。

最终 Demo 应该让评委看到：

> 用户只是正常开了几场会，Agent 就能把个人待办、后续日程、主题知识库、会议总结、转写记录和关联资料组织起来，而且所有关键动作都经过用户确认。这正是从“被动搜索知识”到“主动分发和维护知识”的变化。
